Introduction

01 — Overview

What is DevPulse?

Most developers have no idea what their coding patterns actually look like over time. They know they shipped a feature this week, but they don't know whether they've been getting faster, whether their PR review cycles are shrinking, whether they're accumulating technical momentum — or silently burning out. DevPulse exists to change that.

DevPulse is a developer productivity dashboard that connects to GitHub via OAuth and surfaces meaningful, personal metrics: commit streaks, pull request throughput, code review patterns, contribution heatmaps, and language breakdowns. It aggregates raw GitHub activity into high-signal, actionable insights — then presents them in a way that makes growth visible and, crucially, motivating.

The core insight driving the project: you improve what you measure, and you sustain improvement when measurement feels rewarding. DevPulse applies this principle to software engineering.

⚠ The Problem

  • GitHub's built-in graphs are per-repo, not per-developer
  • No streak tracking or gamified momentum signals
  • No cross-repo aggregation for holistic view
  • No visibility into PR review turnaround or churn patterns
  • Performance reviews rely on anecdote, not data
  • Remote developers have no social layer for engineering activity

✓ The Solution

  • OAuth-connected, always up-to-date GitHub analytics
  • Gamified commit streaks with milestone celebrations
  • Cross-repo contribution heatmap (like GitHub, but yours)
  • PR throughput and review latency trends over time
  • Language breakdown to track skill breadth
  • Weekly digest email — your engineering "Wrapped"

User clicks "Sign in with GitHub"

Frontend navigates to /api/v1/auth/github — a NestJS route that triggers the Passport redirect.

2

Redirect to GitHub Authorization endpoint

3

User reviews & approves scopes

user:email — read profile email · repo — read repository data & commits · read:org — read org membership

4

GitHub redirects to our callback

GET /api/v1/auth/github/callback?code=… — Passport intercepts this route.

5

passport-github2 exchanges code → access token, calls GitHubStrategy.validate()

Passport handles the token exchange with GitHub's OAuth API transparently. Our strategy only sees the final accessToken and passport profile.

6

GitHubProfileVO is constructed

An immutable Value Object wraps the raw passport profile, normalizing nullable fields and encapsulating the access token before it touches domain logic.

7

IdentityService.loginWithGitHub() — upsert & issue JWT

The GitHub access token is AES-encrypted before persistence. The User row is upserted (created on first login, updated on subsequent logins). A short-lived JWT is signed with the user's ID as subject.

8

Redirect to frontend with JWT

302 → http://localhost:3000/auth/callback?token=eyJ…

9

Frontend stores JWT, redirects to /dashboard

The /auth/callback page extracts the token from the query string, persists it to localStorage, and performs a client-side redirect. All subsequent API calls attach Authorization: Bearer <token>.

Auth Implementation

The strategy stays deliberately thin — it only shapes raw Passport data into a well-typed Value Object. All business decisions live in IdentityService. 

// packages/api/src/identity/infrastructure/github.strategy.ts
// GitHubStrategy — Passport strategy, kept intentionally thin
async validate(accessToken: string, _refreshToken: string, profile: Profile) {
  return GitHubProfileVO.fromPassportProfile(profile, accessToken)
}

// packages/api/src/identity/application/identity.service.ts
// IdentityService.loginWithGitHub — the actual auth use-case
async loginWithGitHub(profile: GitHubProfileVO) {
  // 1. Encrypt the GitHub token before it ever touches the database
  const encryptedToken = await this.encryptionService.encrypt(profile.accessToken)

  // 2. Upsert: create user on first login, update avatar/email on subsequent ones
  const user = await this.userRepository.upsertFromGitHub({
    githubId:       profile.githubId,
    name:           profile.displayName,
    email:          profile.email,
    avatarUrl:      profile.avatarUrl,
    encryptedToken,
  })

  // 3. Issue a short-lived JWT (subject = user.id)
  const accessToken = this.issueJwt(user)
  return { accessToken, user }
}

Domain-Driven Design

03 — DDD Deep Dive

What is Domain-Driven Design?

Domain-Driven Design (DDD) is a software philosophy introduced by Eric Evans that says: the structure of your code should mirror the structure of your business problem, not the structure of your database schema or your framework's folder conventions. The idea is to build a shared language (ubiquitous language) between engineers and domain experts, then encode that language directly into the codebase.

Instead of thinking in tables and HTTP handlers, you think in domains, bounded contexts, aggregates, value objects, and domain events. The infrastructure — Postgres, Redis, NestJS — becomes a detail. The domain becomes the center of gravity.

Bounded Context

An explicit boundary within which a particular domain model is consistent and unambiguous. The word "User" in the Identity context means something different from "User" in a Billing context — bounded contexts prevent these meanings from bleeding into each other.

Aggregate

A cluster of domain objects that are treated as a single unit for data changes. Each aggregate has a root entity — the only object external code can hold a reference to. All mutations go through the root. This enforces invariants and prevents inconsistent state.

Value Object

An immutable object that has no identity of its own — two Value Objects with the same attributes are considered equal. Examples: GitHubProfileVO, EmailAddress, DateRange. They are described by what they are, not who they are.

Domain Event

A record of something meaningful that happened in the domain — past tense, immutable. UserConnectedGitHub, StreakBroken. Other contexts consume these events without being coupled to the producer's internals.

Repository Port

An interface (port) that the domain layer depends on to persist and retrieve aggregates — without knowing how they are stored. The Prisma implementation is an adapter that satisfies this interface. The domain never imports Prisma directly.

Anti-Corruption Layer

A translation layer at the boundary between two contexts. When the Analytics context needs to know if a user exists, it goes through an ACL that translates Identity's User into Analytics' own concept — preventing vocabulary contamination.

Notice what a Value Object is not: it has no id, no database row, no save() method. All fields are readonly. The factory method fromPassportProfile normalizes nullable passport fields in one place — no other code needs to guard for undefined from Passport's raw profile. 

// packages/api/src/identity/domain/github-profile.vo.ts
export class GitHubProfileVO {
  constructor(
    readonly githubId:     string,
    readonly username:     string,
    readonly displayName:  string | null,
    readonly email:        string | null,
    readonly avatarUrl:    string | null,
    readonly accessToken:  string,
  ) {}

  /**
   * Factory method: the ONLY place that knows about Passport's raw profile shape.
   * All nullable coercion is centralised here, never scattered across the codebase.
   */
  static fromPassportProfile(profile: Profile, accessToken: string): GitHubProfileVO {
    return new GitHubProfileVO(
      profile.id,
      profile.username ?? profile.id,          // fallback to numeric id if username absent
      profile.displayName ?? null,
      profile.emails?.[0]?.value ?? null,       // first verified email, or null
      profile.photos?.[0]?.value ?? null,       // avatar URL, or null
      accessToken,
    )
  }
}

The DailyMetrics decision

Storing pre-aggregated daily rows instead of raw commit events is a deliberate trade-off. A raw events table grows by thousands of rows per active user. Querying "commits in the last 90 days across 20 repos" requires scanning potentially millions of rows. With DailyMetrics, the same query touches at most 90 days × 20 repos = 1,800 rows — O(range × repos), constant regardless of commit frequency.

How Contexts Communicate

Today (pragmatic)

Direct service injection inside the NestJS DI container. When the Notifications context needs to know a user's display name, it calls an IdentityQueryService that exposes only a minimal read model — not the full Identity service with all its mutation methods.

This is appropriate at the current scale: one process, one deployment unit, fast iteration. The discipline comes from respecting the boundaries in code — never importing across contexts' internal folders.

Tomorrow (event-driven)

BullMQ is already wired in for the sync jobs. The planned evolution is to route domain events through BullMQ queues between contexts — same Redis instance, but now each context is a consumer of events it cares about.

At scale this becomes a true event bus (or even Kafka), but the code structure doesn't change — only the transport layer. That's the payoff of drawing the boundaries correctly now.

Anti-Corruption Layer (ACL) pattern: Each context exposes a thin public facade to other contexts. The Analytics context doesn't consume IdentityService directly — it uses an IdentityReadModel interface that translates Identity concepts into terms Analytics understands. This means a breaking change inside Identity (e.g., renaming displayName to name) propagates no further than the ACL adapter.

Primary Ports Driven by outside world

Why Daily Granularity?

Not hourly

24× more rows for resolution that no dashboard chart uses. The heatmap, streak, and throughput charts all operate on day-level data.

Not raw events

Storing 1 row per commit would require aggregation on every query. DailyMetrics pre-aggregates during sync — reads are instant.

repoId = null dual-use

Rows with repoId=null are pre-aggregated user totals. One query gives the "all repos" overview card without a SUM() over all repos.

Computed fields (not stored — derived at query time):
netLines = additions - deletions — net contribution to codebase size
churnRatio = deletions / (additions + deletions) — high churn (>0.6) signals refactor days

Aggregation + Cache Pattern

// Redis-cached, 5-minute TTL — BullMQ sync worker busts cache on completion
async getDashboardMetrics(userId: string, from: string, to: string) {
  const cacheKey = `analytics:dashboard:${userId}:${from.slice(0, 10)}:${to.slice(0, 10)}`

  return this.redis.getOrSet(
    cacheKey,
    () => this.metricsRepo.getDailyMetrics(userId, from, to),
    300,  // 5-minute TTL
  )
}

// Called by SyncRepositoryProcessor after writing metrics to DB
async invalidateDashboardCache(userId: string) {
  const pattern = `analytics:dashboard:${userId}:*`
  await this.redis.deleteByPattern(pattern)
}
Why this works: GitHub sync is async (BullMQ job), so the DB is stable between syncs. The 5-minute TTL ensures freshness on the first load after a sync, while cache hits absorb all repeated dashboard refreshes in between.

Encrypted Token Storage — AES-256-GCM

// AES-256-GCM: encrypt-then-authenticate in one pass
async encrypt(plaintext: string): Promise<string> {
  const key = await this.getKey()          // derived from JWT_SECRET via scrypt (expensive)
  const iv = randomBytes(12)           // fresh random IV per encryption
  const cipher = createCipheriv('aes-256-gcm', key, iv)
  const encrypted = Buffer.concat([
    cipher.update(plaintext, 'utf8'),
    cipher.final(),
  ])
  const authTag = cipher.getAuthTag()  // GCM: authenticate ciphertext integrity

  // Stored as "iv:authTag:ciphertext" — all hex-encoded
  return [iv.toString('hex'), authTag.toString('hex'), encrypted.toString('hex')].join(':')
}
Random IV

Encrypting the same token twice produces different ciphertext. Prevents frequency analysis — an attacker can't detect when two users have the same token.

Auth Tag (GCM)

GCM = encrypt + authenticate in one pass. The 16-byte auth tag proves the ciphertext was not tampered with. Decryption throws if the tag doesn't match.

scrypt KDF

The AES key is derived from JWT_SECRET via scrypt (memory-hard). Even if the DB is compromised, brute-forcing the key requires significant CPU + RAM per attempt.

10.1

Repository Pattern

Problema

A lógica de negócio (AnalyticsService, StreakService) não deve conhecer Prisma, SQL, ou qualquer ORM. Se precisar trocar de banco, o domínio não deve ser tocado.

10.2

Guard Pattern (NestJS)

O AuthGuard('jwt') do NestJS funciona automaticamente para HTTP. Para GraphQL, o contexto de execução é diferente — precisamos de uma subclasse que ensine o guard onde encontrar o request.

gql-auth.guard.ts 
@Injectable()
export class GqlAuthGuard extends AuthGuard('jwt') {
  // AuthGuard('jwt') funciona para HTTP out-of-the-box
  // Para GraphQL precisamos sobrescrever getRequest()
  override getRequest(context: ExecutionContext) {
    const ctx = GqlExecutionContext.create(context)
    return ctx.getContext<{ request: unknown }>().request
  }
}
Breakdown linha a linha
extends AuthGuard Herda toda a lógica de validação JWT do Passport — só precisamos mudar como o request é extraído
ExecutionContext Polimórfico — pode ser HTTP, WebSocket ou GraphQL. O NestJS usa o mesmo tipo para todos
GqlExecutionContext Converte o contexto genérico para a variante GraphQL, expondo o contexto Apollo onde o Passport coloca o usuário após validação JWT
.request O request HTTP que chegou com o JWT no header Authorization — onde o Passport vai procurar o Bearer token
APLICAÇÃO

Decorado no nível da classe: @UseGuards(GqlAuthGuard) — toda query e mutation daquele resolver exige autenticação. Uma linha protege dezenas de endpoints.

10.4

Strategy Pattern (Passport)

Passport é nativamente baseado no padrão Strategy. Cada estratégia encapsula um algoritmo de autenticação diferente. O DevPulse registra duas: 'github' para OAuth e 'jwt' para validação de token.

github.strategy.ts 
@Injectable()
export class GitHubStrategy extends PassportStrategy(Strategy, 'github') {
  constructor(configService: ConfigService) {
    super({
      clientID: configService.getOrThrow('GITHUB_CLIENT_ID'),
      scope: ['user:email', 'repo', 'read:org'],
    })
  }
  async validate(accessToken, _refreshToken, profile): Promise<GitHubProfileVO> {
    // Chamado pelo Passport após OAuth callback bem-sucedido
    // Retorna um Value Object normalizado, não o perfil bruto do Passport
    return GitHubProfileVO.fromPassportProfile(profile, accessToken)
  }
}
Duas estratégias, dois fluxos
'github' Strategy

Acionada no redirect OAuth. Recebe o token de acesso do GitHub, valida o perfil, normaliza para GitHubProfileVO. Roda uma vez no login.

'jwt' Strategy

Acionada em cada request autenticado. Extrai e valida o Bearer token. Popula req.user com o payload JWT decodificado.

  // Factory method — único ponto de criação do VO
  static fromPassportProfile(profile, accessToken): GitHubProfileVO {
    return new GitHubProfileVO(
      String(profile.id),
      profile.username,
      profile.displayName ?? null,
      profile.emails?.[0]?.value ?? null,
      profile.photos?.[0]?.value ?? null,
      accessToken,
    )
  }
}
IMUTABILIDADE

Todos os campos são readonly. Um VO criado não pode ser modificado — apenas substituído por outro.

SEM IDENTIDADE

Nenhum campo id. Dois VOs com os mesmos valores são iguais por definição — a identidade vem dos dados, não de uma chave artificial.

ANTI-CORRUPTION

O factory method normaliza o perfil bruto e inconsistente do Passport antes de entrar no domínio. Camada anti-corrupção no nível do objeto.

10.6

Factory Method + Static Constructor

GitHubProfileVO.fromPassportProfile() é um factory method estático. O perfil retornado pelo Passport tem campos opcionais, aninhados em arrays, com tipagem fraca. O factory normaliza esse formato imprevisível em uma forma confiável antes de qualquer código de domínio tocá-lo.

CONCEITO

O construtor privado (implícito via factory) garante que não existe outro caminho para criar um GitHubProfileVO sem passar pela normalização. Impossível criar um VO mal-formado acidentalmente.

Cache-Aside Pattern

A aplicação controla o cache manualmente: tenta ler do Redis, se não existir vai ao banco, persiste o resultado. Esse controle explícito é mais previsível do que caching automático transparente.

redis.service.ts + analytics.service.ts 
// Utility genérica no RedisService
async getOrSet<T>(key: string, factory: () => Promise<T>, ttlSeconds: number): Promise<T> {
  const cached = await this.get<T>(key)
  if (cached !== null) return cached    // cache hit
  const fresh = await factory()           // cache miss: executa a factory
  await this.set(key, fresh, ttlSeconds)
  return fresh
}

// Uso no AnalyticsService
return this.redis.getOrSet(
  `analytics:dashboard:${userId}:${from}:${to}`,
  () => this.metricsRepo.getDailyMetrics(userId, from, to),
  300  // TTL: 5 minutos
)

// Invalidação após sync — wildcard zera todos os ranges do usuário
await this.redis.delPattern(`analytics:dashboard:${userId}:*`)

factory Função lazy — só executada em caso de cache miss. Evita chamadas desnecessárias ao banco.
<T> Genérico — o chamador recebe o tipo correto de volta. O Redis armazena JSON; o get<T> faz o parse e retorna tipado.
delPattern O wildcard * invalida todos os ranges de datas de um usuário de uma vez. Sem isso, um sync novo serviria dados antigos do cache.

Highlight 01

StreakCalculator.calculate() — o coração da gamificação

Pure Function

Função estática e pura que converte uma lista desordenada de datas em streak atual e longest streak. Zero dependências externas — testável com vi.fn() zero vezes.

static calculate(activeDates: Date[]): StreakResult {
  if (activeDates.length === 0)
    return { currentStreak: 0, longestStreak: 0, lastActiveDate: null }

  // Deduplicar e ordenar como strings YYYY-MM-DD
  const uniqueDays = [...new Set(activeDates.map(d => this.toDateKey(d)))].sort()

  let longestStreak = 1, currentRun = 1

  for (let i = 1; i < uniqueDays.length; i++) {
    const prev = new Date(uniqueDays[i - 1])
    const curr = new Date(uniqueDays[i])
    const diffDays = Math.round((curr.getTime() - prev.getTime()) / 86_400_000)
    if (diffDays === 1) { currentRun++; if (currentRun > longestStreak) longestStreak = currentRun }
    else currentRun = 1
  }

  // O streak ainda está ativo? Última atividade deve ser hoje ou ontem
  const today = this.toDateKey(new Date())
  const yesterday = this.toDateKey(new Date(Date.now() - 86_400_000))
  const lastDay = uniqueDays.at(-1)

  let currentStreak = 0
  if (lastDay === today || lastDay === yesterday) {
    currentStreak = 1
    for (let i = uniqueDays.length - 2; i >= 0; i--) {
      const diff = Math.round((new Date(uniqueDays[i+1]).getTime() - new Date(uniqueDays[i]).getTime()) / 86_400_000)
      if (diff === 1) currentStreak++
      else break
    }
  }

  return { currentStreak, longestStreak: Math.max(longestStreak, currentStreak), lastActiveDate: new Date(lastDay) }
}
new Set() + .sort()

Múltiplos commits no mesmo dia colapsam em um dia ativo. Sort em strings YYYY-MM-DD funciona corretamente pois lexicográfico = cronológico.

86_400_000

Milissegundos por dia. O separador numérico _ (ES2021) melhora a legibilidade sem custo. Math.round() protege contra horário de verão (±1h).

Dois passes

1° passe (forward): encontra longestStreak em toda a série histórica. 2° passe (backward): conta o currentStreak a partir do dia mais recente.

today || yesterday

Se você commitou ontem e agora é meia-noite, o streak não é quebrado ainda. Aceitar "ontem" é essencial para a UX — o usuário não perde o streak ao dormir.

Highlight 02

AES-256-GCM — Criptografia do access token

Security

O token de acesso do GitHub é criptografado antes de armazenar no banco. Se o banco vazar, os tokens permanecem inacessíveis sem a chave derivada do JWT_SECRET.

async encrypt(plaintext: string): Promise<string> {
  const key = await this.getKey()        // scrypt(JWT_SECRET, 'devpulse-v1', 32 bytes)
  const iv = randomBytes(12)             // IV de 96 bits — padrão para GCM
  const cipher = createCipheriv('aes-256-gcm', key, iv)
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()])
  const authTag = cipher.getAuthTag()    // tag de autenticação de 128 bits
  return [iv.toString('hex'), authTag.toString('hex'), encrypted.toString('hex')].join(':')
}
scrypt Derivação de chave baseada em senha — computacionalmente caro por design. Resistente a brute-force mesmo se o banco vazar. Produz 32 bytes = chave AES-256.
randomBytes(12) IV fresco a cada criptografia. Mesmo plaintext gera ciphertext diferente a cada chamada. Sem isso, padrões repetidos seriam detectáveis.
GCM mode Galois/Counter Mode: AES simétrico (confidencialidade) + GHASH (autenticação) em um único passo. Mais seguro que CBC que não autentica o ciphertext.
getAuthTag() Tag de integridade. Se o ciphertext for modificado no banco, a decriptografia lança um erro — detecta tampering antes de processar dados corrompidos.
iv:tag:cipher Tudo em uma coluna string — sem colunas extras no schema. O IV e a tag precisam estar junto ao ciphertext para decriptografia.

Highlight 06

churnRatio com exactOptionalPropertyTypes

TypeScript

Com exactOptionalPropertyTypes: true, { churnRatio: undefined } é um erro de tipo porque churnRatio?: number significa "ausente" — não "presente com valor undefined".

private mapMetrics(m: DailyMetrics): DailyMetricsType {
  const total = m.additions + m.deletions
  const base = {
    id: m.id, date: m.date, commits: m.commits,
    additions: m.additions, deletions: m.deletions,
    prsOpened: m.prsOpened, prsMerged: m.prsMerged,
    reviewsDone: m.reviewsDone, netLines: m.additions - m.deletions,
  }
  // chave omitida inteiramente quando não há dados — não "undefined"
  return total > 0 ? { ...base, churnRatio: m.deletions / total } : base
}
KEY INSIGHT

O spread condicional total > 0 ? { ...base, churnRatio: X } : base é o padrão correto. Quando total === 0 a chave churnRatio não existe no objeto retornado. O TypeScript em strict mode detectaria { ...base, churnRatio: undefined } como erro de compilação.

Apollo Client v4 — auth link com breaking change

v4 Breaking

Apollo Client v4 mudou a assinatura do setContext: o segundo argumento agora é prevContext e a função deve retornar o contexto completo, não um patch.

const authLink = setContext((_op, prevContext) => {
  const token = getToken()
  const prev = prevContext as { headers?: Record<string, string> }
  return {
    headers: {
      ...(prev.headers ?? {}),                              // preservar headers existentes
      ...(token ? { authorization: `Bearer ${token}` } : {}), // só adicionar se token existe
    },
  }
})
Apollo v3 (antigo)
setContext((_, { headers }) => ({
  headers: {
    ...headers,
    authorization: token
      ? `Bearer ${token}`
      : ''
  }
}))
Apollo v4 (correto)
setContext((_op, prevContext) => ({
  headers: {
    ...(prev.headers ?? {}),
    ...(token
      ? { authorization: `Bearer ${token}` }
      : {})  // ausente — não "undefined"
  }
}))
DETALHE CRÍTICO

O spread ...(token ? { authorization: ... } : {}) garante que o header authorization fica completamente ausente quando não há token — não presente com valor "Bearer undefined", que causaria 401 silencioso em rotas públicas.

Como Escalar

De monolito funcional para 100K usuários — gargalos reais, decisões arquiteturais concretas.

Gargalos atuais (DevPulse hoje)

GitHub API Rate Limit

5.000 req/hora por usuário. Irrelevante em pequena escala. Crítico quando 1 usuário rastreia 50 repos com syncs horários: cada sync consome ~100 requests.

redis.keys(pattern) — O(N)

Usado no delPattern(). Bloqueia o servidor Redis para varrer todas as chaves. Aceitável em < 10K usuários. Degrada linearmente acima disso.

Prisma findMany sem select

Retorna objetos completos do banco. Em alto volume, colunas não usadas (como campos de auditoria) são transmitidas desnecessariamente em cada query.

Single BullMQ worker

Um processo de worker processa jobs sequencialmente. Funcional para < 1.000 syncs simultâneos. Acima disso, a fila cresce indefinidamente.

Event Sourcing

Eventos brutos ao invés de upsert por dia

O Que Eu Aprendi

Reflexões honestas — não uma lista de fatos, mas as trade-offs reais que emergiram no processo.

1

GraphQL foi a decisão certa para este dashboard

Inicialmente considerei REST. Mas um dashboard que consulta "commits + PRs + heatmap para o range de datas X" é um caso de uso perfeito para GraphQL — a consulta tem forma exatamente como o tipo de dado. O schema tipado capturou várias incompatibilidades de contrato entre frontend e backend durante o desenvolvimento. O tipo MetricsRangeInput tornava impossível chamar metrics() sem especificar datas — a API força o contrato em tempo de compilação.

TRADE-OFF QUE NAO ESPERAVA

Apollo Client v4 foi uma quebra significativa. Os subpath imports (@apollo/client/link/context), a nova assinatura do setContext, e o genérico removido ApolloClient<object> causaram horas de confusão — a maioria dos tutoriais ainda mostram a API v3.

2

Arquitetura Hexagonal: over-engineering no início, correto em retrospecto

Na primeira versão, todas aquelas interfaces e símbolos de injeção pareciam cerimônia desnecessária. Mais código por mais código. Então escrevi os primeiros testes — e entendi: mockar IGitHubPort levou 3 linhas de vi.fn(). Testar o StreakCalculator não precisou de nenhum mock — função pura, sem dependências.

INSIGHT

As interfaces (ports) forçam direção de dependência clara — domínio não depende de infraestrutura. Essa clareza não se paga na escrita, se paga nos testes e na primeira vez que você precisa trocar um adapter. O custo upfront é real; o benefício é diferido mas composto.

3

Prisma 7 foi uma armadilha que não vi chegando

O Prisma 7 silenciosamente removeu o campo url do datasource em schema.prisma. Agora ele vive em prisma.config.ts junto com a configuração do driver adapter. O guia de migração existe, mas a mensagem de erro — "Expected string, got undefined" — não aponta para ele. Perdi tempo considerável debugando uma mudança de configuração que deveria ser trivial.

RECOMENDACAO

Ficaria no Prisma 6 por mais 6 meses até o ecossistema do 7.x estabilizar. A mudança do driver adapter traz benefícios reais de performance, mas o custo de migração e a escassez de exemplos atualizados não compensam para projetos novos em 2025.

4

Pré-agregar em DailyMetrics foi a granularidade certa

O design inicial previa armazenar commits brutos e agregar nas queries. Então calculei: 100 commits/dia × 365 dias × 20 repos = 730K linhas só para um usuário ativo — e cada query de heatmap precisaria agregar tudo isso. Pré-agregar em uma linha por dia por repo significa que a query do dashboard é O(range_days × repos) — rápida mesmo em escala, sem agregação em query time.

CUSTO DA DECISAO

Pré-agregação é uma aposta: você escolhe quais métricas importam antecipadamente. Se amanhã o usuário quiser saber "commits feitos antes das 9h" ou "mensagens de commit com mais de 100 caracteres", a informação está perdida. Event sourcing preservaria a optionalidade ao custo de mais storage e complexidade de query.

5

BullMQ com deduplicação resolveu um problema de UX que não havia antecipado

Sem jobId, clicar "Sync" rapidamente enfileirava 10 jobs idênticos. O primeiro completava e invalidava o cache; os 9 restantes faziam trabalho idêntico, consumiam rate limit do GitHub, e confundiam o usuário com o estado de loading intermitente. Uma linha — jobId: `sync:${repositoryId}` — eliminou o problema inteiro. É o tipo de detalhe que você só descobre testando o fluxo real, não pensando sobre ele em abstrato.

O que eu faria diferente

1
Começar com testes

Escrever IMetricsRepository primeiro e a implementação Prisma depois força design de interface melhor — a interface emerge das necessidades do domínio, não da API do ORM. Escrever testes post-hoc revelou que algumas interfaces tinham granularidade errada.

2
Event sourcing desde o dia 1

DailyMetrics pré-agregadas são práticas mas com perdas. Não é possível responder "qual dia teve o maior commit antes das 9h?" ou "commits com mensagem de mais de 50 palavras". Eventos brutos preservariam toda a optionalidade analítica futura.

3
Schema-first GraphQL para o contrato compartilhado

A abordagem code-first trata o SDL como artefato gerado. Schema-first deixa o time de frontend definir as queries que precisa e o backend as implementa. Melhor para times — a interface é o contrato, não o código TypeScript.

4
Indicador de rate limit do GitHub na UI

A implementação atual trata erros de rate limit graciosamente mas não os exibe. Um indicador "GitHub API: 4.832/5.000 remanescentes, reset em 43 min" geraria confiança e explicaria proativamente por que um sync pode demorar.

5
Zustand foi over-engineering para o estado atual

Zustand foi adicionado para o estado de colapso da sidebar. useState + context teria bastado. Adicionar "para uso futuro" é exatamente o tipo de abstração prematura que o projeto deveria evitar. O código paga custo de complexidade sem entregar benefício.

"DevPulse nasceu como um exercício em construir algo real com padrões modernos de TypeScript. O stack é opinado. A arquitetura é deliberada. O resultado mais valioso não é o dashboard — é o instinto para tomar essas decisões sob incerteza."
